Fix: if `parentElement.getBoundingClientRect().width > container rect width`, use the parent width (and temporarily set `container.style.width` so foreignObject layout matches).

    New Vitest jsdom test (TC-PDF-005): construct a container where `container.getBoundingClientRect()` returns `{width: 200, height: 300}` but `container.parentElement.getBoundingClientRect()` returns `{width: 800, height: 300}`. Also assert that after capture, `container.style.width` is restored to its original value (empty string).

    Also add a reverse-case assertion in the same test or a sibling: if parent width ≤ self width, the original self width is used (don't shrink widgets that are already correctly sized).
  </behavior>

    2. GREEN: In `domToImageCapture`, replace the width/height resolution block (lines 106-112) with:
    ```ts
    const selfRect = container.getBoundingClientRect();
    const parentRect = container.parentElement?.getBoundingClientRect();
    const useParent = parentRect && parentRect.width > selfRect.width;
    const width = Math.max(1, Math.round(useParent ? parentRect.width : selfRect.width));
    const height = Math.max(1, Math.round(selfRect.height));

    // When using parent width, temporarily stretch the container so
    // foreignObject layout matches the allocated tile width. Availability
    // Finder and other fit-content widgets shrink to intrinsic width
    // otherwise. Restored in finally.

    if (useParent) {

    try {

        cacheBust: true,
        width,
        height,
        canvasWidth: width,
        canvasHeight: height,
        filter: (node) => { /* filter from Task 1 — unchanged */ },
      });
    } finally {
      if (useParent) {
        container.style.width = prevInlineWidth;
      }
    }
    ```
