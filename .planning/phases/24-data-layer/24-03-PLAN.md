- `month_series` from sixMonthsAgo to currentMonth

- Per-entity (department or person), per-month utilization:
  - If groupBy='department': GROUP BY department_id, department_name, month
  - If groupBy='person': GROUP BY person_id, name, month (LIMIT to top `limit` by avg utilization DESC)

Build two separate SQL queries based on groupBy (use conditional SQL template, same approach as the filter pattern: `const query = groupBy === 'department' ? sql\`...\` : sql\`...\``).

Transform:
- For each entity: build months Record (YYYY-MM -> utilization %)
- currentUtilization = value for current month
- Get value for 6 months ago. changePercent = currentUtilization - sixMonthsAgoUtilization (absolute difference, not relative)
- direction: 'up' if changePercent > 2, 'down' if < -2, 'stable' otherwise
- isOverloaded = currentUtilization > 100
- If groupBy='department': include headcount (count of people in that department)

**2. `getPersonSummary(orgId, personId: string): Promise<PersonSummaryResponse>`**

Three data regions in a single function (can be 1-2 queries):

a) **Person info**: SELECT person data with discipline + department joins. Validate person belongs to org and is not archived. Throw ValidationError if not found.

b) **Current month + 6-month trend**:
- Current month = `new Date().toISOString().slice(0, 7)`
- 6-month window ending at current month
- Query: per-month utilization for this person, with per-project breakdown for current month

SQL:
```sql
WITH month_series AS (
  SELECT to_char(d, 'YYYY-MM') AS month
  FROM generate_series(sixMonthsAgo::date, currentMonth::date, '1 month')
)
SELECT ms.month, a.project_id, pr.name AS project_name, COALESCE(a.hours, 0)::int AS hours
FROM month_series ms
LEFT JOIN allocations a ON a.person_id = personId::uuid
  AND to_char(a.month, 'YYYY-MM') = ms.month
  AND a.organization_id = orgId::uuid
LEFT JOIN projects pr ON pr.id = a.project_id
ORDER BY ms.month, hours DESC
```

c) **Upcoming availability** (next 3 months after current):
Same query pattern but for months currentMonth+1 through currentMonth+3.

Transform:
- currentMonth: totalAllocated = sum of project hours, utilizationPercent = totalAllocated/target*100, available = max(0, target - totalAllocated)
- status: 'empty' if totalAllocated=0, 'overloaded' if utilization>100, 'warning' if utilization>85, 'healthy' otherwise
- projects: array with projectId, projectName, hours, percentOfTarget = hours/target*100, color from CHART_PALETTE
- trend.months: Record of YYYY-MM -> utilization% for last 6 months
- trend.changePercent and direction: same logic as utilization-trends
- upcomingAvailability: array of {month, available} for next 3 months

**API Routes:**

`utilization-trends/route.ts`: Extract groupBy param (validate as 'department'|'person', default 'department'). Extract optional limit (parse number, default 10). No from/to needed (fixed 6-month window).

`person-summary/route.ts`: Extract personId param (required, validate non-empty). No from/to needed. Throw ValidationError if personId missing.

**3. `getProgramRollup(orgId, monthFrom, monthTo, programId?: string): Promise<ProgramRollupResponse>`**

SQL structure — most complex query, needs multiple JOINs:

a) **Program info** (if programId provided):
```sql
SELECT pg.id, pg.name, COUNT(DISTINCT pr.id) AS project_count
FROM programs pg
LEFT JOIN projects pr ON pr.program_id = pg.id AND pr.organization_id = orgId::uuid
WHERE pg.id = programId::uuid AND pg.organization_id = orgId::uuid
GROUP BY pg.id, pg.name
```
If no programId, set program to null (all programs mode).

b) **Monthly load + people**: For projects in the program (or all projects if no programId):
```sql
WITH program_projects AS (
  SELECT pr.id, pr.name, pr.status
  FROM projects pr
  WHERE pr.organization_id = orgId::uuid
    AND (programId IS NULL OR pr.program_id = programId::uuid)
    AND pr.archived_at IS NULL
),
month_series AS (...standard...),
project_allocations AS (
  SELECT pp.id AS project_id, pp.name AS project_name, pp.status,
         ms.month,
         COUNT(DISTINCT a.person_id)::int AS people_count,
         COALESCE(SUM(a.hours), 0)::int AS total_hours
  FROM program_projects pp
  CROSS JOIN month_series ms
  LEFT JOIN allocations a ON a.project_id = pp.id
    AND to_char(a.month, 'YYYY-MM') = ms.month
    AND a.organization_id = orgId::uuid
  GROUP BY pp.id, pp.name, pp.status, ms.month
)
```

c) **Discipline coverage**: CROSS JOIN all org disciplines with program people. For each discipline: count people who have that discipline and are allocated to program projects. Coverage = people_with_discipline / total_disciplines_needed (where needed = at least 1 for each discipline). Use simpler approach: coveragePercent = if people allocated with this discipline > 0 then min(100, allocatedHours / avgTargetHours * 100) else 0. gapFte = if coveragePercent < 80 then (1 - coveragePercent/100) * expectedPeopleCount else 0.

Transform:
- monthlyLoad: Record of YYYY-MM -> total hours across all program projects
- totalPeople: COUNT DISTINCT people allocated to any program project
- peakMonthlyHours: max value from monthlyLoad
- staffingCompleteness: percentage of disciplines with coveragePercent >= 80%
- projects: per-project summary with peopleCount (avg across months), monthlyHours (avg), status
- gapAlert: if any discipline has coveragePercent < 50%: "Critical gap in {discipline} — {gapFte} FTE needed"

```sql
  LEFT JOIN allocations a ON ... AND to_char(a.month, 'YYYY-MM') = ms.month ...
  GROUP BY ap.department_id, ap.department_name
),
period_b AS (
  -- same for period B
  ...
)
SELECT ...
```

Transform:
- periodA/B labels: auto-generate from date ranges, e.g., "Jan-Mar 2026" or "Q1 2026"
- Compute 5 comparison metrics:
  1. "Average Utilization" — overall utilization % for each period
  2. "Headcount" — active people count
  3. "Total Allocated Hours" — sum across period
  4. "Overloaded People" — count with utilization > 100%
  5. "Bench Hours" — sum of (target - allocated) for people with utilization < 80%
- For each metric: delta = valueB - valueA, deltaPercent = delta/valueA*100, signal = 'improving' if utilization closer to 85%, 'worsening' if further, 'neutral' if change < 2%

- notableChanges: auto-generated bullet strings for significant changes (> 10% delta)
