**1. `getAvailabilityTimeline(orgId, monthFrom, monthTo, filters?: { departmentId?, disciplineId?, availableOnly? }): Promise<AvailabilityTimelineResponse>`**

SQL structure (extends the getTeamHeatMap pattern but adds per-project breakdown):
- `month_series` CTE as standard
- `active_people` CTE: JOIN departments and disciplines, apply optional departmentId/disciplineId filters, same WHERE pattern as getTeamHeatMap
- `person_month_projects` query: for each person in each month, get per-project allocation hours. SQL:
  ```sql
  SELECT ap.person_id, ap.first_name, ap.last_name, ap.target_hours,
         ap.department_id, ap.department_name, ap.discipline_abbreviation,
         ms.month,
         a.project_id, pr.name AS project_name, COALESCE(a.hours, 0)::int AS hours
  FROM active_people ap
  CROSS JOIN month_series ms
  LEFT JOIN allocations a ON a.person_id = ap.person_id
    AND to_char(a.month, 'YYYY-MM') = ms.month
    AND a.organization_id = orgId::uuid
  LEFT JOIN projects pr ON pr.id = a.project_id
  ORDER BY ap.department_name, ap.last_name, ap.first_name, ms.month
  ```

Transform rows:
- Group by department -> person -> month -> projects array
- For each person-month: totalAllocated = sum of project hours, available = max(0, target - totalAllocated), utilizationPercent = totalAllocated/target * 100 (0 if target is 0)
- Assign project colors using CHART_PALETTE[projectIndex % CHART_PALETTE.length] — maintain consistent project-to-color mapping across all people
- If `availableOnly` filter: exclude people who have 0 available hours across ALL months in range
- Summary: totalAvailableHours = sum of all available across all people/months, peopleWithAvailability = count of people with any available month

**2. `getAvailabilitySearch(orgId, monthFrom, monthTo, filters?: { disciplineId?, departmentId?, minHours?, sort? }): Promise<AvailabilitySearchResponse>`**

SQL structure:
- `month_series` + `active_people` (JOIN departments + disciplines for names)
- Per-person per-month utilization query (similar to above but without project breakdown)
- Dynamic sort clause (use the pattern from research):

  const sortClause = {
    available: sql`total_available DESC`,
    utilization: sql`avg_utilization ASC`,

Transform:

- totalAvailable = sum of available across months
- avgUtilization = average utilization across months
- If minHours filter: exclude people with totalAvailable < minHours
- Return total count of matching results

`availability/route.ts`: Extract from/to, optional disciplineId, departmentId, minHours (parse as number: `Number(params.get('minHours')) || undefined`), sort (validate as 'available'|'utilization'|'name', default 'available').

**3. `getBenchReport(orgId, monthFrom, monthTo, threshold = 80): Promise<BenchReportResponse>`**

SQL structure — use a SINGLE query with two period CTEs to avoid two separate queries (per research performance note):
- `month_series` for requested period
- `prev_month_series` for previous period of equal length (compute: if range is N months, previous = N months before fromDate)
- `active_people` CTE: JOIN departments + disciplines
- `person_utilization` CTE: per-person, compute avg utilization % across requested period months
- `prev_utilization` CTE: same computation for previous period
- Benched = people with utilization < threshold%

Transform:
- Filter to benched people (utilization < threshold)
- `totalBenchHours` = for each benched person: (target - allocated) summed across months in range
- `fteEquivalent` = totalBenchHours / (monthCount * average target_hours_per_month across all benched people)
- Group by department: sum bench hours, count people, compute FTE per department
- Group by discipline: same
- `topAvailable`: top 10 benched people sorted by freeHoursPerMonth DESC
- `trendVsPrevious`: compute previousBenchHours from prev_utilization CTE, changePercent = ((current - previous) / previous * 100), direction = 'up' if change > 2%, 'down' if < -2%, 'stable' otherwise
- `insight`: if benchHours increased > 10% from previous: "Bench capacity increased by {pct}% — consider redistributing {topDept} resources"

**4. `getConflicts(orgId, month?, months = 1): Promise<ConflictsResponse>`**

Params: `month` defaults to current month (format YYYY-MM from `new Date()`), `months` = how many months forward to check (1-12).

SQL structure:
- `month_series`: generate_series from month to month + (months-1) months
- `active_people` CTE
- Per person, per month: total allocated hours. Filter to people where totalAllocated > target_hours.
- Also get per-project breakdown for conflicted people.

Query (two-step approach):
1. First query: find conflicted people + their total allocations per month
2. The per-project breakdown can be embedded in the same query using a subquery or fetched separately. Use same query with project join:
   ```sql
   SELECT ap.person_id, ..., ms.month, a.project_id, pr.name AS project_name,
          COALESCE(a.hours, 0)::int AS project_hours,
          SUM(COALESCE(a.hours, 0))::int OVER (PARTITION BY ap.person_id, ms.month) AS month_total
   FROM active_people ap
   CROSS JOIN month_series ms
   LEFT JOIN allocations a ON a.person_id = ap.person_id
     AND to_char(a.month, 'YYYY-MM') = ms.month AND a.organization_id = orgId::uuid
   LEFT JOIN projects pr ON pr.id = a.project_id
   ```
   Then filter in TypeScript to only people where month_total > target_hours.

Transform:
- For each conflicted person-month: overBy = totalAllocated - target
- projects array: all projects with hours for that person-month
- suggestedResolution: sort projects by hours DESC, suggest reducing the largest by overBy amount. `{ projectId, projectName, reduceBy: overBy, newHours: largestHours - overBy }`. Only if newHours >= 0.
- summary.totalConflicts = count of unique conflicted people
- summary.resolvedThisMonth = 0 (no resolution tracking yet, placeholder)

**API Routes:**

`bench-report/route.ts`: Extract from/to via validateMonthRange. Extract optional threshold (parse as number, default 80, clamp 0-100).

`conflicts/route.ts`: Extract optional month param (validate with MONTH_RE pattern: `/^\d{4}-\d{2}$/`, default to current month via `new Date().toISOString().slice(0, 7)`). Extract optional months param (parse as number, default 1, clamp 1-12). Do NOT use validateMonthRange here since this endpoint uses month+months instead of from/to.
