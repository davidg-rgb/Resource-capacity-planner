    Build the onboarding wizard as a set of 'use client' components. Follow the existing import wizard pattern (wizard-stepper.tsx) for step navigation UX.

    **onboarding-wizard.tsx** -- Main orchestrator:
    - 'use client' component
    - State: `step` (0-3 index for departments/disciplines/people/complete)
    - On mount: fetch `GET /api/onboarding/status` to check if already partially set up. Also fetch `GET /api/departments`, `GET /api/disciplines`, `GET /api/people` to get existing items for idempotency (per Pitfall 2 in research).
    - Render a step indicator (styled similar to wizard-stepper.tsx) showing: 1. Departments, 2. Disciplines, 3. People, 4. Done
    - Each step gets: `onNext` callback to advance, `onSkipAll` callback to skip entire wizard
    - `onSkipAll`: calls `POST /api/onboarding/complete`, then `router.push('/input')` (per ONBR-05)
    - Wrap in a centered card layout with heading "Set up your workspace"
    - Use Tailwind classes consistent with app design: bg-surface, rounded-lg, shadow, p-6/p-8

    **step-departments.tsx** -- Department creation:
    - Show `DEPARTMENT_SUGGESTIONS` as clickable chip buttons (per ONBR-03)
    - Clicking a suggestion chip calls `POST /api/departments` with that name, adds to local added list, disables the chip
    - Show already-existing departments (from initial fetch) as "Already added" chips (greyed out)
    - Text input + "Add" button for custom department names
    - "Next" button to advance to disciplines
    - "Skip all" link text at bottom of every step

    **step-disciplines.tsx** -- Discipline creation:
    - Show `DISCIPLINE_SUGGESTIONS` as clickable chip buttons with abbreviation labels
    - Clicking a suggestion calls `POST /api/disciplines` with name + abbreviation
    - Show already-existing disciplines as greyed chips
    - Text inputs for custom name + abbreviation + "Add" button
    - "Next" and "Skip all" buttons

    **step-people.tsx** -- First person creation:
    - Simple form: name (required), select department (from those just created), select discipline (from those just created)
    - Calls `POST /api/people` on submit
    - Show any already-existing people count: "You already have N people"
    - Offer alternative: "Or import from spreadsheet" link to `/import` (calls onboarding/complete first)
    - "Next" and "Skip" buttons

    **step-complete.tsx** -- Success screen:
    - Show checkmark icon and "You're all set!" message
    - Summary: "N departments, M disciplines, P people created"
    - Call `POST /api/onboarding/complete` automatically on mount
    - Two CTA buttons: "Go to Resource Planning" (link to /input), "View Team Overview" (link to /team-overview)
    - Use Lucide icons (CheckCircle, ArrowRight)

    Error handling for all steps:
    - Wrap fetch calls in try/catch
    - Show toast (sonner) on API errors: `toast.error('Failed to create department')`
    - Duplicate name errors (409 from unique constraint): show inline "Already exists" instead of toast

    2. After org creation (detected via Clerk's `afterCreateOrganizationUrl`), render the `OnboardingWizard` component
    3. Implementation approach: Use Clerk's `useOrganization()` hook to detect if an org is active. If no active org, show CreateOrganization. If org exists, show OnboardingWizard.

    ```typescript
    'use client';

    import { CreateOrganization, useOrganization } from '@clerk/nextjs';
    import { OnboardingWizard } from '@/components/onboarding/onboarding-wizard';

    export default function OnboardingPage() {
      const { organization, isLoaded } = useOrganization();

      if (!isLoaded) return <LoadingSkeleton />;

      if (!organization) {
        return (
          <div className="bg-surface flex min-h-screen items-center justify-center">
            <CreateOrganization afterCreateOrganizationUrl="/onboarding" />
          </div>
        );
      }

      return (
        <div className="bg-surface flex min-h-screen items-center justify-center p-4">
          <OnboardingWizard />
        </div>
      );
    }
    ```

    Key: `afterCreateOrganizationUrl` redirects back to `/onboarding` (not `/input`), so the wizard is shown after org creation. The wizard's completion flow handles navigation to `/input`.
