# Phase 52 Plan 02: LM-03 Queue Count Endpoint Summary

**One-liner:** LM-03 ships `GET /api/v5/proposals/queue/count` + `getQueueCount` service fn — per-tenant, per-department live-dept-routed count of `proposed` proposals; 11 PGlite-backed tests green (5 service + 6 route).

Wave 1's critical backend artifact: the count source that Plan 52-04 will consume for the LM approval-queue badge on `/line-manager` and the persona-switcher count suffix. Ships the service function, the route handler, and 11 integration tests. Threat-model mitigations T-52-01 / T-52-02 / T-52-04 are all exercised by dedicated tests; T-52-03 / T-52-05 / T-52-06 / T-52-07 are accepted or naturally satisfied by the shape of the endpoint.

## What shipped

### 1. `getQueueCount` service function (D-05)

**File:** `src/features/proposals/proposal.service.ts` (appended after `resubmitProposal`, before the Plan 39-03 approve/reject block).

**Signature:**
```ts
export async function getQueueCount(orgId: string, departmentId: string): Promise<number>
```

**SQL (as generated by drizzle):**
```sql
SELECT count(*)::int AS "count"
FROM "allocation_proposals"
INNER JOIN "people"
  ON "allocation_proposals"."person_id" = "people"."id"
WHERE "allocation_proposals"."organization_id" = $1
  AND "allocation_proposals"."status" = 'proposed'
  AND "people"."department_id" = $2
```

**Filter contract (locked, mirrors `listProposals`):**
- `organization_id = orgId` — T-52-04 tenant isolation
- `status = 'proposed'` — **single value, NOT `inArray([...])`**, per Pitfall #7 in RESEARCH; `counter_proposed` does not exist in the enum
- `people.department_id = departmentId` — PROP-07 live-dept routing (NOT the snapshot column `target_department_id`)

**Return:** `Number(row?.count ?? 0)` — unboxes the drizzle `sql<number>` expression; `0` when no rows match.

### 2. `GET /api/v5/proposals/queue/count` route handler (D-05)

**File:** `src/app/api/v5/proposals/queue/count/route.ts` (new)

**Request:** `GET /api/v5/proposals/queue/count?departmentId=<uuid>`

**Response:**
- `200 { count: number, departmentId: string }` — authenticated, valid UUID
- `400 { error: { code: 'ERR_VALIDATION', ... } }` — missing or malformed `departmentId`
- `401 { error: { code: 'ERR_AUTH', ... } }` — no auth (Clerk rejection)
- `403 { error: { code: 'ERR_FORBIDDEN', ... } }` — role below `planner`

**Threat-model mitigations wired:**
| Threat | Mitigation in route |
|--------|---------------------|
| T-52-01 Spoofing | `requireRole('planner')` — Clerk gate |
| T-52-02 Tampering | Zod `z.string().uuid()` on `departmentId` |
| T-52-04 Info Disclosure (DB) | Delegates to service which filters by `orgId` |
| T-52-05 Info Disclosure (logs) | No custom logging of `departmentId`; `handleApiError` wraps ZodError without echoing the attempted value beyond zod's issue message (which names the field, not the payload) |

**Not flag-gated:** Per `52-CONTEXT.md` open_questions Q6, this endpoint is always live; `uiV6PerJourney` is read only by UI consumers.

### 3. Service contract tests (5 tests)

**File:** `src/features/proposals/__tests__/proposal.service.queue-count.test.ts` (new, 348 LOC)

| Test | Seeds | Expected |
|------|-------|----------|
| happy path | 3 `proposed` in D1/A | `getQueueCount(A, D1)` → 3 |
| status filter exactness | 3 proposed + 2 approved + 1 rejected + 1 withdrawn + 1 superseded (all D1/A) | → 3 (only `proposed`) |
| department scope | 3 `proposed` in D1/A + 2 in D2/A | `(A, D1)` → 3, `(A, D2)` → 2 |
| tenant isolation | 3 `proposed` in D1/A + 2 in D1/B | `(A, D1_B)` → **0**; `(B, D1_B)` → 2 |

Fixture pattern mirrors `proposal.service.create.test.ts`: PGlite + drizzle; 2 orgs, 3 depts, 7 people, 2 projects; `beforeEach` truncates `allocation_proposals` so org/dept/people rows are stable. Raw `INSERT INTO allocation_proposals` (no `createProposal` / `changeLog` noise) keeps the test surface minimal.

### 4. Route integration tests (6 tests)

**File:** `src/app/api/v5/proposals/queue/__tests__/count.test.ts` (new, 298 LOC)

| Test | Behavior asserted |
|------|-------------------|
| 200 happy | authenticated planner + valid UUID → `{ count, departmentId }` |
| 400 missing | no `departmentId` query param → `ERR_VALIDATION` |
| 400 malformed | `departmentId=not-a-uuid` → `ERR_VALIDATION` |
| 401 unauth | `requireRole` throws `AuthError` → 401 `ERR_AUTH` |
| **tenant isolation (route level)** | auth as org A, request dept of org B → 200 `count=0`; sanity: auth as B returns 2 |
| state-filter round-trip | 3 `proposed` + 2 `approved` + 1 `withdrawn` in D1/A → 200 `count=3` |

Mutable `fakeAuth` stub supports both unauth (`throw new AuthError`) and cross-tenant (`orgId` swap). `requireRole` + `getTenantId` mocks honor the flag. No `departmentId` values are logged by the tests themselves (T-52-05 conformance).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] `node_modules` missing in fresh worktree**
- **Found during:** first `pnpm test` invocation
- **Issue:** `'vitest' is not recognized` — worktree's dependencies not installed.

- **Files modified:** none (lockfile respected, no package.json edits).

### Decisions taken on behalf of the plan

- **`new URL(request.url).searchParams` vs `request.nextUrl.searchParams`:** Plan's verbatim skeleton used `request.nextUrl.searchParams`. First GREEN attempt returned 500 on every case except `401 unauth` because the test harness passes a vanilla `Request` (not a `NextRequest`), and `request.nextUrl` is populated by the Next.js pipeline. Switched to `new URL(request.url).searchParams.entries()` to match the pattern already used in the neighboring `src/app/api/v5/proposals/route.ts`. Behavior is identical in production Next.js runtime (both return the same `URLSearchParams`); the edit unblocked the test harness without changing the route contract.
- **Cross-tenant response = silent 0 (not 403):** The plan's tenant-isolation test asserts `200 { count: 0 }`, not `403`. Kept that contract — rationale in frontmatter `decisions`. A 403 would leak the existence of the foreign-tenant dept id. Counting 0 does not.

### Scope calls

No out-of-scope fixes were applied. The plan's success criteria and acceptance greps all pass cleanly on top of 52-01's baseline. No pre-existing failures surfaced that required Rule-3 triage.
