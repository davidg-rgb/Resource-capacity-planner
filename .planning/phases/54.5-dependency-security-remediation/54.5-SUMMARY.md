# Phase 54.5: Dependency security remediation

**Took `pnpm audit` from 2 critical / 14 high / 14 moderate / 3 low (33 vulns) to FULLY CLEAN (0/0/0/0). The `dependency_audit` execution gate — which BLOCKed during Phase 54 and was overridden — now passes on its own.**

## Why this phase existed
Phase 54's execution gates fired live at production tier: `dependency_audit` BLOCKed on
33 pre-existing transitive vulnerabilities and was overridden (`54-GATES.md`) because
Phase 54 changed zero dependencies. The override was the record that this remediation was
owed. This pass pays it off.

## Approach
Planned, incremental, tested after every change — not a blind `pnpm update`. Each bump was
its own atomic commit with `pnpm install -> typecheck -> lint -> test (vitest, 1094) -> pnpm audit`
before moving to the next. Order was by blast radius: auth first, then framework, then ORM,
then app-facing moderates, then dev-tooling transitive moderates.

## What was bumped (and what each cleared)

| Commit | Bump | Cleared |
|--------|------|---------|
| `382e11e` | `@clerk/nextjs` 7.0.7 -> 7.4.2 | **2 critical** (middleware redirect bypass) + **4 high** (org-role auth bypass) + js-cookie high (transitive) |
| `8b61bbe` | `next` 16.2.1 -> 16.2.6 (+ eslint-config-next) | **8 high** (DoS ×3, SSRF, 4× middleware/proxy bypass) + 4 moderate (XSS ×2, image DoS, RSC cache poisoning) + 2 low (cache) |
| `37a8cba` | `drizzle-orm` 0.45.1 -> 0.45.2 | **1 high** — SQL injection via improper escaping (multi-tenant ORM) |
| `43ae35c` | `next-intl` 4.8.3 -> 4.12.0, `uuid` -> 13.0.2 | 2 moderate (open-redirect, prototype-pollution) + 1 moderate (uuid buffer bounds) + 1 low (icu-minify) |
| `013a9df` | pnpm.overrides: postcss/brace-expansion/esbuild | 4 moderate (postcss XSS, brace-expansion ReDoS ×2, esbuild dev-server leak) |
| `9dca2e4` | `vitest` 2.1.9 -> 3.2.4 + vite ^6.4.2 override | 1 moderate (vite optimized-deps path traversal) |

After the first three bumps `pnpm audit --audit-level=high` was already CLEAN (the gate-clearing
checkpoint). The last three swept the moderates/low to zero.

## Verification at every step
- `pnpm typecheck` — green throughout
- `pnpm lint` (eslint + mutations-manifest) — green throughout
- `pnpm test` — **1094 passing** after every bump (only the pre-existing `imports.api`
  env-harness suite fails — a Phase 58 / QUAL-04..06 dev-env gap, identical to the baseline,
  NOT introduced here)
- `pnpm audit` — final state: `{critical:0, high:0, moderate:0, low:0}`
- exec-gates engine: `parse-audit` -> `shouldFail:false`; `decide --gate dependency_audit` -> verdict `pass`
- Auth path (highest risk): `requireRole`/`getTenantId` (`auth()` -> userId/orgId/orgRole) and
  `clerkMiddleware`/`createRouteMatcher`/`auth.protect()` surface unchanged; typecheck + the
  rbac/audit contract suites confirm the `{orgId, userId, role}` contract still holds
- drizzle-kit config still transpiles under esbuild >=0.25 (`drizzle-kit check` -> "Everything's fine")

## Deviations / notable calls
- **vite needed a vitest major bump.** vite >=6.4.2 is the patched line; vitest 2 pins vite ^5,
  so vitest had to go 2 -> 3.2.4 (supports vite 6). A `vite@<6.4.2 -> ^6.4.2` override holds it
  on the 6.x line (an unpinned `>=6.4.2` floated it to vite 8, which vitest 3.2.4 does not support).
- **3 snapshot regenerations.** vite 6 hashes CSS-module class names differently than vite 5
  (`_cell_e1c43c` -> `_cell_b6907e`). DOM/attrs/values are byte-identical — pure hash drift.
  Production CSS is compiled by Next/turbopack, not vite, so this hash only ever existed in the
  vitest/jsdom test env. Regenerated, not chased as a regression.

## Known follow-ups (non-blocking)
- `environmentMatchGlobs` is deprecated in vitest 3 (removed in v4). Migrate `vitest.config.ts`
  to `test.projects` before any future vitest 4 bump. Out of scope for a security pass.
- `regression` override in `54-GATES.md` stays until **Phase 58** fixes the `imports.api`
  env-harness (QUAL-04..06).
