## 1. Executive Summary

Nordic Capacity v5.0 turns the existing single-view resource planner into a **persona-aware planning-and-actuals platform**. Four overlapping views (Project Manager, Line Manager, Staff, R&D Manager) share one underlying plan/actual dataset. The two defining additions are:

1. **Actuals layer** — a new day-grain `actual_entries` table, plus an Excel-first import pipeline with idempotent re-import, batch rollback, and an override-manual-edits checkbox.
2. **Proposal → approval workflow** — PMs submit planning *wishes* that live in a separate `allocation_proposals` table alongside the approved `allocations` table; line managers approve, reject, or (optional) counter-propose; everything lands in a universal `change_log`.

Three supporting pillars:

- **Role switcher** (header control) drives scope and default landing for each persona. No auth is added — personas are UX shortcuts, not security boundaries.
- **ISO 8601 / 53-week first-class** — v5.0 ships into 2026 (53-week year). Week handling is centralized in one utility module; no JS-native date defaults allowed anywhere.
- **Change log as audit spine** — every edit (PM wish, line manager direct edit, historic correction, import batch) writes a `change_log` row. This doubles as the history trail for editable past periods.

This architecture is strictly **additive** to the existing v4.0 code. The current `allocations` table (monthly grain) remains the source of truth for approved plans. New tables bolt on without migrations that touch existing data.

### 2.2 User Roles & Journeys

Five roles exist in v5.0 (chosen via header switcher, not authenticated):

1. **Project Manager (PM)** — scoped to their project(s). Owns wishes. Edit gates: own project historic = warn, own project future = direct; other-group resources = proposal only.
2. **Line Manager (Linjechef)** — scoped to their line group (= existing `departments` table). Direct edit within own group. Owns approval queue.
3. **Staff (Medarbetare)** — scoped to self. Read-only.
4. **R&D Manager (R&D-chef)** — no scope filter. Read across all projects and groups. Can drill-down everywhere. Edit disabled for v5.0 (delegates to line managers via call/in-person).
5. **Admin** — no scope filter, full edit, owns register-maintenance screens. Functionally a superset of R&D Manager + edit.

All five personas are selectable from a single dropdown. The switcher also picks the *actor* (e.g., "Per — Electronics Design" as Line Manager) because the persona-scoped views need a concrete entity to filter by.

### 2.3 Constraints

- **Tech stack locked** — Next.js 16 / React 19 / Drizzle ORM / Neon Postgres / Clerk (auth exists but unused for personas) / TanStack Query / Tailwind v4 / next-intl / ag-grid / recharts / xlsx (SheetJS) / Zod.
- **Existing data preserved** — `allocations` table and all current v4.0 screens must keep working. v5.0 adds tables and routes; it does not rewrite the monthly-grain plan storage.
- **Scale** — ~20 people × ~10 projects × ~30 months ≈ 6,000 allocation rows per tenant. Actuals at day grain ≈ 20 × 10 × ~250 work days/year ≈ 50–70k rows/year. Small enough that straightforward SQL (no materialized views) is fine.
- **Locale** — Swedish primary; ISO 8601 weeks mandatory; 2026 is a 53-week year.
- **Desktop primary** — mobile is acceptable degradation, not a design target.
- **Launch gate** — PDF export bug (html2canvas blank for non-SVG widgets) is separately tracked; not blocking v5.0 planning work but must be fixed before launch.

### 2.4 Integrations

- **Excel (.xlsx) — primary import source.** SheetJS (`xlsx`) already in dependencies. No API integration to external time systems in v5.0.
- **CSV — secondary import source.** Same parser pipeline, different adapter.
- **Clerk — auth (existing).** Remains in place but unused for persona selection. The role switcher stores persona state in a UI-only context; no changes to Clerk identity.

### 2.5 Resolved Ambiguities

All Q1–Q6 decisions from `.planning/v5.0-FEEDBACK.md` are treated as binding. Two items remain genuinely open and are handled as design-time decisions:

| Open item | Decision timing | Default assumed by this blueprint |
|---|---|---|

| 53-week visual handling in long-horizon view | Phase 4, during R&D Portfolio grid build | Week 53 gets its own column; adjacency to week 1 of next year is visual only, not a merge. Quarter aggregation snaps on ISO-year boundary. |

## 3. Architecture Decision Records

### ADR-001 — Separate `allocation_proposals` table (not a state column on `allocations`)

**Decision.** Pending PM wishes live in a new `allocation_proposals` table. The `allocations` table remains the single source of approved truth.

**Alternatives considered.**
- (a) Add `state` column (`draft`/`proposed`/`approved`/`rejected`) to `allocations`. Rejected because: a single row can't represent "current approved 20h *and* a pending proposal for 60h" at the same time, and the journeys require both to render side-by-side.
- (b) Use the existing `scenarios` machinery for wishes. Rejected because: scenarios are what-if containers (multi-row snapshots) and have no per-row approval lifecycle; conflating them harms both features.

**Consequences.** Two tables to query for the PM view. Upside: `allocations` stays clean and v4.0 reads don't change. Downside: proposal-aware UI components must join both tables; addressed by a dedicated read model in `src/features/planning/planning.read.ts`.

### ADR-002 — Day-grain `actual_entries` table, monthly-grain `allocations` unchanged

**Decision.** Actuals are stored at day grain in a new `actual_entries` table keyed `(organization_id, person_id, project_id, date)`. Approved plans stay monthly grain in `allocations`.

**Alternatives considered.**
- (a) Migrate `allocations` to day grain too. Rejected — unnecessary work, existing v4.0 screens are built around monthly cells.
- (b) Store actuals at month grain. Rejected — Q2 decision; import dumps are per-day.

**Consequences.** Comparison views aggregate actuals up to month on read. Drill-down drawer reads raw daily rows. Aggregation logic lives in `src/features/actuals/actuals.read.ts`.

### ADR-003 — Universal `change_log` table, single writer helper

**Decision.** A single `change_log` table captures every state-changing operation: approved allocation edits, proposal lifecycle events, import batches, register-maintenance actions. All writes go through one helper function `recordChange(...)` so no code path can bypass the log.

**Alternatives considered.**
- (a) Per-table audit columns (`updated_by`, `previous_hours`). Rejected — doesn't capture *why* or *how* (proposal? direct edit? import?), and can't represent multi-row operations like a batch rollback.
- (b) DB triggers. Rejected — harder to test, harder to debug, obscures intent from TypeScript code.

**Consequences.** Every mutating service function must call `recordChange` in the same transaction. Enforced by code review + an eslint-style test that asserts every mutating service emits a change log entry (see §15 checklist).

### ADR-004 — Role switcher is UI-only React context; no server-side role enforcement

**Decision.** Persona selection lives in a client-side React context (`PersonaContext`) persisted to `localStorage`. Server routes ignore persona — they enforce tenant scoping via Clerk org ID (existing) and nothing more.

**Alternatives considered.**
- (a) Pass persona to server and enforce. Rejected — Q1 explicitly said personas are not security boundaries, and adding server-side enforcement doubles the work without any real protection.
- (b) Full role-based access control with Clerk metadata. Rejected — adds permanent complexity for a temporary UX shortcut.

**Consequences.** Any API route is callable with any persona. This is acceptable given the single-tenant demo deployment model. If multi-tenant security ever matters, a v6.0 auth phase replaces this without touching UI code (UI reads its current persona from the same context; only the source of truth shifts from localStorage to a server session).

**Round 2 clarifications:**
- **Persona switcher dropdown is fully open** — lists all five persona types regardless of who is logged in. Anyone can pick any persona. Tested in TC-PSN-010.
- **R&D persona is UI-read-only**, even though server-side does not enforce it. Every screen reachable from the R&D landing route renders timeline grids with `onCellEdit` omitted; no Approve/Reject buttons; no Submit Wish path. This is enforced at the component-prop level via a wrapper `<ReadOnlyScope persona='rd'>` that the page layouts apply. Tested in TC-RD-READONLY-001 and TC-NEG-014.
- **Staff persona** is also UI-read-only for the same reasons (already covered by TC-NEG-001).
- **Mobile desktop-only routes** wrap their page content in a `<DesktopOnlyScreen>` component that, on viewports <768px, replaces the page body with a friendly "Använd dator för denna vy" interstitial and a back button to the persona's landing route. Routes wrapped: S6, S8, S10. Tested in TC-MOBILE-001.

### ADR-005 — Centralized ISO 8601 / 53-week utility, no inline date math

**Decision.** All week/month/quarter/year logic lives in `src/lib/time/iso-calendar.ts`. No other file may import from `date-fns` directly or use native `Date.getDay()` / `toLocaleDateString()`. An eslint rule enforces this.

**Alternatives considered.**
- (a) Let each feature import `date-fns` as needed. Rejected — too easy to default to US-week semantics by accident. The 2026 53-week year makes this a correctness risk, not a style preference.

**Consequences.** Every date operation is routed through a small API. Tests for this module (§15) block all other date tests.

### ADR-006 — Excel parsing via SheetJS with a two-stage pipeline (parse → stage → commit)

**Decision.** Import proceeds in three explicit states: `staged` (parsed, validated, preview shown) → `committed` (written to `actual_entries` under an `import_batches` row) → optionally `rolled_back`. The staged state is persisted in a `staged_imports` temp table (or reuses the existing `import_sessions` jsonb blob) so the preview survives page reloads.

**Alternatives considered.**
- (a) Single-step import. Rejected — journey 2D requires a preview with override checkbox before commit.
- (b) Client-side-only staging (no DB row). Rejected — user should be able to come back to a preview after a browser crash or accidental tab close.

**Consequences.** Import state machine has four states (`parsing`, `staged`, `committed`, `rolled_back`). Rollback available within 24h window or until next import, whichever comes first.

### ADR-007 — ag-grid for all timeline grids (reuse existing)

**Decision.** All v5.0 timeline grids (PM timeline, Line Mgr group view, Staff schedule, R&D portfolio) use ag-grid community (already a dependency). A thin wrapper `TimelineGrid` component owns the ag-grid instance and exposes the v5.0-specific cell, edit, and drill-down semantics.

**Alternatives considered.**
- (a) Build a custom grid. Rejected — reinventing virtualization, cell editing, keyboard nav is off-scope.
- (b) Use a simpler library. Rejected — ag-grid is already proven in v4.0 and handles the sizes (200+ columns for long-horizon view) without issue.

**Consequences.** v5.0 binds to ag-grid's API. A column definition factory (`buildTimelineColumns`) centralizes all column construction so we can swap the grid library later if needed without touching feature code.

### ADR-008 — Auto-save for direct edits; explicit Submit for proposals

**Decision.** Two distinct save semantics depending on what the user is doing:

1. **Direct edits** (line manager editing their own group, PM editing their own department's people, admin editing anything) — debounced 600ms auto-save via TanStack Query mutation. Optimistic UI shows the pending value immediately. No separate unsaved-changes buffer.
2. **Proposals** (PM editing a person from another department) — the cell switches into "proposal mode" on first keystroke and *does not auto-save*. The user must click **Submit wish**, optionally add a note, and explicitly send. Until they click submit, the value is held in a local component-level state and is discarded if they click away.

This resolves the previous contradiction between client #6 ("auto-save in planning view") and Journey 1B ("clicks Submit wish"). Auto-save applies to the edits the client was actually thinking about (their own day-to-day plan tweaks); proposal submission is deliberate by design because it pulls another person into a workflow.

**Alternatives considered.**
- (a) Auto-save everything. Rejected — proposal submission needs the optional note and an explicit "I'm asking for this" intent.
- (b) Explicit save for everything. Rejected — client #6 explicitly asked for auto-save in normal planning.

**Consequences.**
- Direct edits: network hiccups roll back the optimistic value and show a toast. Asserted in §15.
- Proposals: a "discard unsaved proposal" guard fires if the user navigates away with an unsubmitted draft (component-level dirty check; no server state involved).
- Visual indicator for direct-edit auto-save: subtle "Sparar…" label in the cell during the mutation, fades on success, becomes red "Misslyckades" toast on error with retry.

### ADR-008b — Line manager edit boundary (clarification)

**Decision.** Line manager auto-saves apply if and only if `editedPerson.departmentId === currentLineManager.departmentId`. **Project ownership is irrelevant** to this gate. A line manager editing a person in their own group auto-saves regardless of which PM owns the project that person is allocated to.

This was implicit but not stated in ADR-008/ADR-009; round 2 review caught the ambiguity. Tested in TC-PS-015.

### ADR-009 — PM ↔ department / project ownership stored on existing tables

**Decision.** Persona scoping for PM and Line Manager personas resolves through two new columns on existing tables:
- `people.default_department_id` — *the* department a person primarily belongs to. For Line Managers, this is the group they own. For PMs, this is their "home" department; edits to people in this department are direct, edits to others go through proposal.
- `projects.lead_pm_person_id` (nullable) — the PM who owns a project. Used by `getMyProjects(pmPersonId)` to populate Anna's project picker.

The existing `people.departmentId` column (already on the table at v4.0) is reused as `default_department_id` — no new column. Verified by reading `src/db/schema.ts`: `people.departmentId` already exists and is required. We rename the *concept* for clarity, not the column.

`projects.lead_pm_person_id` is the only schema addition required for persona scoping.

**Alternatives considered.**
- (a) Many-to-many `project_managers` join table. Rejected — v5.0 client mental model is "one PM per project, sometimes a backup." The backup is a v6.0 concern.
- (b) Many-to-many `line_manager_departments`. Rejected — Q5/Q6 conversations confirmed one line manager owns one department in this client's org chart. Documented as a hard constraint.

**Consequences.**
- One nullable column added to `projects` (migration is trivial).
- `getMyProjects(orgId, pmPersonId)` is a single-table query: `SELECT * FROM projects WHERE lead_pm_person_id = ?`.
- The "is this PM editing their own group?" check is `editTarget.person.departmentId === currentPersona.homeDepartmentId`, where `homeDepartmentId` is loaded once when the PM persona is selected.
- If the data has `lead_pm_person_id IS NULL` for a project, that project simply does not appear in any PM's picker — it lives only in line-manager and R&D views. This is acceptable; PM ownership is opt-in.
- See §15 for `TC-PSN-007` and `TC-PR-014` covering the new gating.

### ADR-010 — Distribution rounding: largest-remainder method, deterministic, exact-sum

**Decision.** When distributing N hours across K working days, use the **largest-remainder** method:

1. Compute `quota = N / K` (real number).
2. Initial allocation per day: `floor(quota * 100) / 100` (truncate to 2 decimals — matches `numeric(5,2)` storage).
3. Remainder = `N - sum(initial allocations)` — always non-negative, always less than `K * 0.01`.
4. Distribute the remainder one cent (0.01h) at a time to the days with the largest fractional part of `quota`. Tie-break by day order (earliest day first).

This guarantees:
- **Exact sum**: `sum(rows) == input` to the penny, no floating-point epsilon needed.
- **Deterministic**: same input always produces same row distribution. Reversal payload restores correctly.
- **Stable**: re-running with the same input produces the same rows.

Worked example — distribute 37h across 22 working days:
- quota = 1.6818...; initial = 1.68 per day; sum = 36.96; remainder = 0.04 → 4 days get +0.01.
- Result: 4 days at 1.69, 18 days at 1.68. Sum = 4*1.69 + 18*1.68 = 6.76 + 30.24 = 37.00. ✓

Worked example — 37h across 5 working days:
- quota = 7.4 exactly; initial = 7.40; sum = 37.00; remainder = 0.
- Result: 5 days at 7.40. ✓

**Alternatives considered.**
- (a) Banker's rounding per day. Rejected — non-deterministic on ties for our purposes, and sum drift is possible.
- (b) Float storage + epsilon comparison. Rejected — `numeric(5,2)` is the right type and float introduces sum drift.

**Consequences.**
- TC-AC-004 changed: assert exact equality (`sum(rows) === input`), not epsilon.
- Implemented in `iso-calendar.distribute(totalHours: number, dayCount: number): number[]` — pure function, easy to unit-test exhaustively for inputs 0..1000h × 1..31 days.

### 6.1 Module: `lib/time/iso-calendar.ts`

**Purpose.** Single source of truth for all date, week, month, quarter, and year operations. Nothing else in the codebase may call `date-fns` or native `Date` methods that depend on locale.

```
isoWeek(date: Date): { year: number, week: number }
  Purpose: Return ISO 8601 week number and ISO week-year for a date.
  Params: date — any Date object.
  Returns: { year, week } where year is the ISO week-year (may differ from calendar year for late-Dec / early-Jan dates).
  Side effects: none.
  Error cases: Invalid Date throws ValidationError.
  Called by: timeline-columns, capacity.read, actual.read, change-log.read, anywhere.
  Calls: date-fns getISOWeek, getISOWeekYear.

weeksInIsoYear(year: number): 52 | 53
  Purpose: Return how many ISO weeks the given year contains (53 for 2026, 2032, 2037, 2044; 52 otherwise in scope).
  Params: year — ISO year (1900–2100 supported).
  Returns: 52 or 53.
  Side effects: none.
  Error cases: out-of-range year throws.
  Called by: timeline-columns (long-horizon), R&D portfolio grid.

isoDate(date: Date): string
  Purpose: Format as YYYY-MM-DD (no time component). Used for DB `date` column matching.
  Params: date.
  Returns: ISO date string.

  Purpose: Parse YYYY-MM-DD to Date. Rejects anything else.
  Returns: Date at 00:00 UTC.

  Purpose: Return YYYY-MM for grouping actuals-by-month. Also used as `allocations.month` key.
  Returns: "2026-05".

quarterKey(date: Date): string
  Purpose: Return YYYY-Qn.
  Returns: "2026-Q2".

rangeMonths(start: Date, end: Date): string[]
  Purpose: Enumerate month keys inclusive for a timeline column build.
  Called by: timeline-columns.

rangeWeeks(start: Date, end: Date): { year: number, week: number }[]
  Purpose: Enumerate ISO weeks inclusive. Handles 53-week years.
  Called by: timeline-columns (week zoom).

workDaysInMonth(year: number, monthIndex: number): number
  Purpose: Count Mon–Fri working days in a calendar month, EXCLUDING Swedish public holidays (see swedishHolidays).
  Called by: actual.service.distributeWeekly, actual.service.distributeMonthly.

workDaysInIsoWeek(year: number, week: number): Date[]
  Purpose: Return Mon–Fri dates of an ISO week, EXCLUDING Swedish public holidays. Handles 53-week years (week 53 of 2026 is Dec 28 2026 – Jan 1 2027). Critical: Jan 1 2027 (New Year's Day) IS a Swedish public holiday and is excluded — week 53 of 2026 returns 4 working days (Dec 28–31), not 5.
  Called by: actual.service.distributeWeekly.

swedishHolidays(year: number): Set<string>
  Purpose: Return ISO date strings (YYYY-MM-DD) for Swedish public holidays in the given year. Hardcoded for 2026 through 2030 inclusive (round 2 extension — v5.0 ships into 2026, planning horizon up to 30 months reaches mid-2028, and the 2-year buffer accommodates unforeseen launch slips and 2029+ planning). Years outside range throw ValidationError. v6.0 must replace with a DB-backed `holidays` table before extending further.
  Includes:
    - New Year's Day (Nyårsdagen) — Jan 1
    - Epiphany (Trettondedag jul) — Jan 6
    - Good Friday (Långfredagen) — hardcoded per year (see table below)
    - Easter Monday (Annandag påsk) — hardcoded per year
    - Labour Day (Första maj) — May 1
    - Ascension Day (Kristi himmelsfärdsdag) — hardcoded per year (39 days after Easter Sunday)
    - National Day (Sveriges nationaldag) — Jun 6
    - Midsummer Eve (Midsommarafton) — Friday between Jun 19–25 inclusive (treated as half-day → counts as non-working in v5.0 simplification)

  HARDCODED EASTER-RELATED DATES (verified against Computus):
    | Year | Easter Sun | Good Fri | Easter Mon | Ascension | Midsummer Eve |
    |------|-----------|----------|-----------|-----------|---------------|
    | 2026 | Apr 5     | Apr 3    | Apr 6     | May 14    | Jun 19        |
    | 2027 | Mar 28    | Mar 26   | Mar 29    | May 6     | Jun 25        |
    | 2028 | Apr 16    | Apr 14   | Apr 17    | May 25    | Jun 23        |
    | 2029 | Apr 1     | Mar 30   | Apr 2     | May 10    | Jun 22        |
    | 2030 | Apr 21    | Apr 19   | Apr 22    | May 30    | Jun 21        |
  These values are baked into `swedishHolidays(year)` as a literal lookup. No runtime Computus algorithm — simpler and removes any chance of off-by-one errors. v6.0 must replace with a DB-backed `holidays` table before extending past 2030.
    - Christmas Eve (Julafton) — Dec 24 (treated as non-working per Swedish convention)
    - Christmas Day (Juldagen) — Dec 25
    - Boxing Day (Annandag jul) — Dec 26
    - New Year's Eve (Nyårsafton) — Dec 31 (treated as non-working per Swedish convention)
  Out-of-range year throws ValidationError. v6.0 may extend or move to a holiday calendar table.
  Called by: workDaysInMonth, workDaysInIsoWeek.

classifyHeader(label: string): 'day' | 'week' | 'month' | 'unknown'
  Purpose: Detect the grain of a single pivoted-layout header cell.
  Heuristics (in order):
    - Matches /^\d{4}-\d{2}-\d{2}$/ or any date format parseable by parseIsoDate → 'day'
    - Matches /^[Vv]\.?\s*\d{1,2}(\s*\d{4})?$/ (Swedish week label) or /^W\d{1,2}([-/]\d{4})?$/ (ISO week label, Monday-start) → 'week'
    - Matches Swedish month names ("januari".."december"), English month names, or /^\d{4}-\d{2}$/ → 'month'
    - Sunday-start US-week patterns are detected separately at the parser layer (raw label inspection) and trigger US_WEEK_DETECTED hard-stop before classifyHeader is called.
    - Otherwise → 'unknown'
  Pure function. Round 3 addition — was referenced by the MIXED_GRAIN_PIVOT detection algorithm but never defined.
  Called by: actuals-excel.parser.parseActualsWorkbook.

distribute(totalHours: number, dayCount: number): number[]
  Purpose: Largest-remainder distribution (see ADR-010). Pure function — no dates, no DB.
  Returns: array of length dayCount, sums to totalHours exactly (numeric(5,2) precision).
  Error cases: dayCount <= 0 → ValidationError; totalHours < 0 → ValidationError.
  Called by: actual.service.distributeWeekly, actual.service.distributeMonthly.

formatWeekLabel(year: number, week: number, locale: 'sv' | 'en'): string
  Purpose: Display a week label. Swedish: "v.14", "v.14 2026".
  Called by: timeline-columns label formatter.

currentMonthKey(): string
  Purpose: Month key for today's date. Used for historic-edit threshold and default landing month.
```

```
commitActualsBatch(input: {
  orgId: UUID, importSessionId: UUID,
  overrideManualEdits: boolean,
  overrideUnrolledImports: boolean,    // separate flag for the second-import-of-same-data case
  actorPersonaId: string
}): Promise<ImportBatch>
  Purpose: Write all staged rows from import_sessions.parsed_data into actual_entries under a new import_batches row.

  Conflict detection runs at preview time AND at commit time (defense in depth):
    1. **Manual conflict**: row has `source='manual'`. Skipped unless `overrideManualEdits=true`. Counted in `rowsSkippedManual`.
    2. **Unrolled-batch conflict**: row currently belongs to an earlier `import_batch` whose `rolled_back_at IS NULL` AND that batch is still within its 24h rollback window. Skipped unless `overrideUnrolledImports=true`. Counted in `rowsSkippedPriorBatch`.

  This prevents the reversal-payload corruption case raised in review: a second import that touches the same rows as a still-rollbackable first import would otherwise capture the *first import's* values in the second batch's `reversal_payload`, breaking rollback semantics for both. By default we refuse the conflict; the user can override deliberately.

  When `overrideUnrolledImports=true`, the prior batch is automatically marked `superseded_at = now()` (a new column on `import_batches`) and becomes non-rollbackable. The new batch captures the values that existed *before* the prior batch (chained from `import_batches.reversal_payload`), so the new rollback is still correct.

  Reversal payload format:
    {
      version: 1,
      rows: [
        { person_id, project_id, date,
          prior: { hours, source, import_batch_id } | null   // null = row was new
        },
        ...
      ]
    }

  Returns: ImportBatch { id, rowsInserted, rowsUpdated, rowsSkippedManual, rowsSkippedPriorBatch, warnings }.
  Side effects: INSERT import_batches, UPSERT actual_entries, optional UPDATE prior import_batches.superseded_at, INSERT change_log (one aggregate ACTUALS_BATCH_COMMITTED row per batch with full counts in context — not per daily row).
  Error cases:
    - session status != 'staged' → ConflictError code=SESSION_NOT_STAGED
    - session expired → NotFoundError
  Called by: api/v5/imports/[sessionId]/commit POST.

rollbackBatch(input: { orgId: UUID, batchId: UUID, actorPersonaId: string }): Promise<{ rowsDeleted: number }>
  Purpose: Undo a committed batch. Deletes all actual_entries with importBatchId=batchId and restores any prior values captured in the batch's `reversal_payload` JSONB column. After successful rollback, sets `reversal_payload = NULL` and `rolled_back_at = now()` on the batch row — the payload is meaningful only while the batch is non-rolled-back.

  Round 3 chain-integrity rule: when `commitActualsBatch` later runs and considers chaining from a prior batch, it MUST treat any batch with `rolled_back_at IS NOT NULL` (regardless of whether `reversal_payload` is NULL or stale) as **non-chainable**. Such batches are skipped in the chain walk; the new batch captures pre-state from the next non-rolled-back ancestor, or directly from current actual_entries if no ancestor exists. This prevents the "stale payload reconstructed wrong pre-state" bug raised in round 3 review. Tested in TC-IMP-015.
  Error cases:
    - batch already rolled back → ConflictError code=BATCH_ALREADY_ROLLED_BACK
    - batch not in org → NotFoundError
    - rollback window expired (>24h or superseded by newer batch on same rows) → ConflictError code=ROLLBACK_WINDOW_EXPIRED
  Called by: api/v5/imports/batches/[batchId]/rollback POST.
```

```
  Called by: imports.service.parseActuals.

detectLayout(sheet: Worksheet): 'row-per-entry' | 'pivoted' | 'unknown'
  Purpose: Heuristic — if column headers contain a `date` column, it's row-per-entry; if multiple column headers parse as dates, it's pivoted; otherwise unknown.
  Error: throws if unknown → parser falls back to asking user for layout choice.

parseRowPerEntry(sheet): ParsedRow[]
  Purpose: Expect columns: `person_name`, `project_name`, `date`, `hours`. Case-insensitive. Swedish synonyms accepted:
    - person column: `person`, `namn`, `medarbetare`, `anställd`
    - project column: `projekt`, `projektnamn`
    - date column: `datum`, `dag`
    - hours column: `timmar`, `tid`, `tim`, `h`
  Round 2 review noted "tid" was missing — added.
  Returns: ParsedRow[] where each row has personName, projectName, date (string), hours (number), sourceRow (for error reporting).

parsePivoted(sheet): ParsedRow[]
  Purpose: Expect leftmost columns `person_name`, `project_name`, then N date columns (could be days, ISO weeks, or months). Date header detection via iso-calendar. Emits one ParsedRow per (person, project, date-column).

type ParsedRow = {
  sourceRow: number
  personName: string
  projectName: string
  date: string          // YYYY-MM-DD
  hours: number
  grain: 'day' | 'week' | 'month'
  isoWeek?: { year: number, week: number }
  monthKey?: string
}

type ParseWarning = {
  sourceRow: number
  code: 'BAD_DATE' | 'BAD_HOURS' | 'EMPTY_PERSON' | 'EMPTY_PROJECT' | 'UNKNOWN_COLUMN' | 'AMBIGUOUS_HEADER' | 'HIDDEN_ROWS_SKIPPED' | 'MERGED_CELLS_FORWARD_FILLED'
  message: string
}

// HARD-STOP errors — these abort the parse, NOT warnings:
//   US_WEEK_DETECTED — pivoted layout has any US-week (Sunday-start) header. Importing into an
//                      ISO-week system is data corruption. Parser throws ValidationError code=US_WEEK_DETECTED
//                      with details.detectedHeaders so the user can fix the source file. If a sheet
//                      mixes US-week and ISO-week labels in the same header row, US_WEEK_DETECTED
//                      takes priority over MIXED_GRAIN_PIVOT (hard stop; never proceed).
//   MIXED_GRAIN_PIVOT — pivoted layout has columns of inconsistent grain. Detection algorithm:
//                       1. For each header column, classify via iso-calendar.classifyHeader(label)
//                          which returns 'day' | 'week' | 'month' | 'unknown'.
//                       2. Collect distinct grains across all classified columns (ignoring 'unknown').
//                       3. If size > 1 → throw ValidationError code=MIXED_GRAIN_PIVOT with details.grainsFound.
//                       4. If all 'unknown' → throw UNKNOWN_LAYOUT instead.
//   UNKNOWN_LAYOUT — detectLayout returned 'unknown'. Throws ValidationError code=UNKNOWN_LAYOUT;
//                    the API response includes a hint to the client to ask the user which layout
//                    to assume (post-MVP UI).
```

### 8.2 Internal Module Contracts

The key internal boundary is **"every mutating service must route through `change-log.service.recordChange` inside its transaction."** This is enforced by:

1. Code review convention.
2. A test (§15) that imports every `*.service.ts` file and asserts that each exported mutating function, when called in a mocked transaction, emits at least one change_log call.

Other invariants:

- **No direct `date-fns` imports outside `lib/time/iso-calendar.ts`** — enforced by an eslint `no-restricted-imports` rule.
- **No direct SQL outside `*.service.ts` / `*.read.ts`** — enforced by `no-restricted-imports` on `drizzle-orm/sql`.
- **Persona is never passed to DB queries** — `actorPersonaId` strings are only for change_log context; no query filters on it.

### 11.5 Hard constraints

Documented invariants that the architecture relies on. Violations require a v6.0 redesign.

1. **One Line Manager owns exactly one department.** No multi-department line managers in v5.0. Approval queue, capacity heatmap, and group timeline all assume singular `departmentId` from the persona context. If the client's org actually has multi-dept managers, the admin can add a separate persona entry per department; the change_log captures the actor accurately either way.
2. **One Project Manager per project (`projects.lead_pm_person_id`).** A project is either owned by one PM or has no PM (NULL → invisible to PM persona, visible only to line mgr / R&D / admin).
3. **Personas are not security boundaries** (re-stated from ADR-004). Any persona can hit any API. Server-side enforcement is intentional non-goal.
4. **`change_log` is append-only.** No DELETE, no UPDATE. v6.0 can add a partition / archival strategy if rows balloon, but rows already written are immutable.
5. **Holidays are hardcoded for 2026–2028.** v6.0 must replace `swedishHolidays` with a database-backed calendar before extending the planning horizon further.

## 12. Extensibility Guide

How v5.0 accommodates v6.0+ without rewrites:

1. **Real auth replacing the role switcher.** PersonaContext provider is the only read source of current persona. Swap its implementation from localStorage to server session; consumers don't change.
2. **Task-level granularity under projects.** Add a `tasks` table; `actual_entries` and `allocation_proposals` can add a nullable `task_id` without breaking existing reads (query defaults `task_id IS NULL`).
3. **New persona (e.g., Controller).** Add a case to the Persona union, add route segment, add read helper in planning.read. No changes to existing personas.
4. **Counter-proposal flow (flagged optional in journey 2B).** Add `counter_proposed` to proposal_status enum, add `counterProposeProposal` service function, add button to WishCard, done. Blueprint is structured to allow this as an additive patch.
5. **Per-day planning (move plans from month to day grain).** Requires a new `allocation_days` table alongside `allocations` for coexistence. Read model switches based on feature flag. Not in v5.0 scope but the read layer abstracts the grain.
6. **External time-system integration.** Implement a new parser under `features/imports/parsers/` (e.g., `harvest-api.parser.ts`). The stage→commit pipeline is parser-agnostic.
7. **Notifications (email/Slack).** Add a `notifications` module that subscribes to change_log. Change log is already the event source; no refactor needed.

Extension rule: **add files, don't modify existing ones.** Every new feature should be addable by creating a new module under `features/`, a new route under `app/api/v5/`, and registering any UI entry points in the persona route map.

## 15. Testable Functional Contract (Build Verification Checklist)

Every item here is a single verifiable assertion. Tests map 1:1 to items. Categorized by concern. Items grouped by module to align with phase-level DoD checks.

### 15.1 ISO Calendar (module: `lib/time/iso-calendar.ts`)

**Correctness**
- [ ] **TC-CAL-001** `isoWeek(new Date('2026-01-01'))` returns `{ year: 2026, week: 1 }`. Reasoning (resolved per review): 2026-01-01 is a Thursday; ISO 8601 W01 of any year is the week containing the first Thursday; therefore 2026-01-01 is in W01 2026, and W01 2026 spans Mon 2025-12-29 → Sun 2026-01-04.

### 15.3 Change log service

- [ ] **TC-CL-001** `recordChange` inserts a row with all required fields populated.
- [ ] **TC-CL-002** `recordChange` refuses unknown action values (TypeScript + runtime Zod).
- [ ] **TC-CL-003** When called within a transaction, the row is not visible until commit.
- [ ] **TC-CL-004** When the outer transaction rolls back, the change_log row rolls back too.
- [ ] **TC-CL-005** **Invariant — implementation strategy clarified per review:** the "every mutating service writes change_log" guarantee is enforced by THREE complementary mechanisms, all of which must be in place. None of them is a single "magic" jest test:
  1. **Eslint custom rule `nordic/require-change-log`** — AST-walks `features/**/*.service.ts`. For each exported async function whose name matches `/^(create|update|delete|edit|submit|approve|reject|commit|rollback|upsert|archive|withdraw|bulk[A-Z])/`, assert the function body contains at least one `recordChange(` call OR is annotated `@no-change-log` (escape hatch with mandatory comment explaining why). Lints in CI; failing the rule fails the build.
  2. **Decorator helper `withChangeLog(action, fn)`** — utility wrapper exported from `change-log.service.ts` that takes a service function and an action name, executes the function inside a transaction, and writes the change_log entry. Adoption is opt-in but encouraged for new services. Removes the burden of remembering to call `recordChange` manually.
  3. **Runtime test `tests/invariants/change-log.coverage.test.ts`** — for each mutating service function listed in `tests/invariants/mutations.json`, call it with mocked deps and assert ≥1 `recordChange` call. The manifest is **auto-generated by a separate codegen script** `scripts/generate-mutations-manifest.ts`, NOT by the eslint rule (round 3 correction: eslint rules cannot legally write files as a side effect). The codegen script walks the AST, finds matching exports, writes the manifest. It runs as a `pnpm prepare` step locally and as a pre-lint CI gate. CI runs the codegen script first, then `git diff --exit-code tests/invariants/mutations.json` — if the manifest changed and was not committed, CI fails with a clear "run pnpm generate locally and commit the result" message. The eslint rule's job is purely to flag missing `recordChange` calls inside service files; it does not touch the manifest.
  Combination = build-time eslint guarantee + auto-generated manifest (codegen script) + runtime test.

## 16. Test Data & Seeding Strategy

The E2E tests in §15.13 reference specific personas and projects. This section pins the fixture so every E2E test starts from an identical, deterministic database state. Implemented in `drizzle/seed.v5.ts` (separate from existing v4.0 seed; can be loaded independently).

Use the standard `uuid` npm package's `v5` export — do NOT hand-roll SHA1 truncation. Round 2 review caught the original under-specification.

### 16.8 Test database isolation

- Each test file calls `await loadV5Seed()` in `beforeAll`, which truncates v5 tables and reseeds.
- Tests must NOT depend on order — each test asserts the seeded state at the start, mutates, asserts, then cleans up via the next file's `loadV5Seed`.

### 16.9 Time mocking

Several tests reference "current date 2026-04-07". This is achieved by:
1. Server-side: `iso-calendar.currentMonthKey()` reads from `process.env.NC_TEST_NOW` if set; fallback is real `Date`.
2. Test setup writes `NC_TEST_NOW=2026-04-07T10:00:00Z` before booting the test server.
3. DB `now()` is harder to mock — historic-edit tests explicitly use the env var via the `iso-calendar.isHistoricPeriod(monthKey, nowMonthKey)` two-argument form, which the service calls inside its transaction. The DB `now()` source remains the production path; tests inject the override.

This is the contract: production = DB clock; tests = env override.
