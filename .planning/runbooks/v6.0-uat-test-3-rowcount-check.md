## Goal

For each migration: confirm that the audit query returns **>0 rows** before apply (proves the migration has work to do, or 0 if production is already clean), then **0 rows** after apply (proves the change took), and **0 rows** after a second idempotent re-run (proves idempotence).

## Pre-flight

1. Connect to the production Neon DB via the Neon console SQL editor (or `psql $PROD_DATABASE_URL` if you've got direct access — production URL lives in Vercel env vars under `DATABASE_URL`).
2. Verify the connection is to the correct project (it should be the prod tenant's branch, not a preview branch — check `current_database()` and the Neon project name in the console).
3. Open a transaction window. **All migrations should run in a single transaction so a partial failure rolls everything back.** Neon supports this via the SQL editor's "Run as transaction" toggle, or wrap each `UPDATE` in `BEGIN; ... COMMIT;`.

## Audit queries (run BEFORE apply)

If any returns >0:

## Idempotence check

## Rollback

Each migration has no built-in rollback (it's a destructive jsonb rewrite). Rollback options:

1. **Neon point-in-time recovery (preferred)** — Neon retains 7-day branch history. From the Neon console, create a branch from a timestamp just before the migration ran, then promote it. Coordinate with any in-flight writes — this is a full DB rollback, not a per-row revert.
2. **LEGACY_LAYOUTS code path** — the `uiV6.polish` flag-off code path renders the pre-trim layout from `LEGACY_LAYOUTS` in `src/features/dashboard/default-layouts.ts`. Setting the flag to `false` per-tenant restores pre-v6.0 behavior **for default layouts**. Custom tenant layouts saved in `dashboard_layouts` will remain stripped — only Neon PITR recovers those.
3. **Manual JSONB patch** — for a single tenant, `UPDATE dashboard_layouts SET layout = $1::jsonb WHERE id = $2` with the pre-migration `layout` value (only possible if you saved it before applying).
