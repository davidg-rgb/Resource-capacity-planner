# v5.0 — Persona User Journeys

**Date:** 2026-04-07
**Companion to:** `v5.0-FEEDBACK.md`
**Purpose:** Screen-by-screen narrative walkthroughs for each persona. These are the "consumer stories" that drive view design and are the source of truth for what screens must exist in v5.0.

**Scope note:** Personas are UX shortcuts (role-switcher in header), not security boundaries. The job-to-be-done is "open the app, land on the view that matches my role, zero hunting."

---

## Persona 1 — Project Manager (Projektledare)

**Name used in journey:** Anna
**Role:** PM for "Projekt Nordlys" (a multi-month engineering project, ~8 people)
**Primary questions:**
- Is my project on track — planned vs actual?
- Do I have the people I need in the months ahead?
- How do I get more hours from the line managers when I need them?

### Journey 1A — Monday morning check-in

**Context:** Anna opens the app on Monday morning to see how last week went and what's coming up.

1. **Land on PM Home.** Role switcher shows "Projektledare — Anna". Header already scoped to her.
2. **Sees a project picker** (her 1–N projects — most PMs have 1, some have 2–3). Nordlys is selected by default (most recent).
3. **Project overview card** at top shows:
   - Total planned hours (this month / this quarter / project lifetime)
   - Total actual hours logged against this project so far
   - Plan-vs-actual delta with color cue (green under, red over)
   - Current month's "burn rate" vs planned
4. **Timeline view below** — horizontal month columns (current month ±N), rows are people assigned to Nordlys. Each cell shows planned hours. Cells where actuals exist show a small second number or split bar (plan | actual).
5. Anna spots March is red — actuals ran 40h over plan for Erik. She clicks the cell.
6. **Drill-down drawer** opens from the right: shows daily breakdown for Erik in March, split plan vs actual. She sees Erik logged 10h on March 15 when nothing was planned.
7. She closes the drawer. Mental note: ask Erik about March 15.

**Key views touched:** PM Home, Project timeline, Person-month drill-down.

### Journey 1B — Submitting a planning wish

**Context:** Anna knows Nordlys needs 60h from Sara (who's in a different line group) in June. Sara is currently allocated 20h to Nordlys in June.

1. From the project timeline, Anna clicks Sara's June cell (currently shows "20h").
2. **Edit mode opens inline.** She types "60". Because Sara's department is not Anna's own, the cell switches to *proposal mode* instead of auto-saving. A small banner appears: *"You are proposing a change to Sara's plan. Sara's line manager (Per) must approve before this goes live."*
3. She can optionally add a note: "Need extra capacity for integration phase — critical for Q3 milestone."
4. Clicks **"Submit wish"** — proposals require an explicit submit (auto-save is only for direct edits within your own scope). The cell now renders with a dashed border and muted color, with a "Pending" badge. A second line shows the current approved value (20h) next to the proposed value (60h).
5. Anna sees a small toast: "Wish submitted. Per will be notified in his approval queue."
6. She can see all her pending wishes in a **"My Wishes"** tab/panel, with status: `proposed` / `approved` / `rejected`.

**Key views touched:** Inline edit with approval gate, My Wishes panel, Pending state visualization.

### Journey 1C — A rejected wish

**Context:** Per rejected the 60h wish with reason "Sara has another commitment — can offer 40h max."

1. Anna opens the app, sees a badge on "My Wishes": 1 rejected.
2. Opens the panel. The Sara/June wish is marked rejected with Per's reason inline.
3. She can **edit and resubmit** from the rejected card — changes 60 → 40, adds a note acknowledging, resubmits.
4. The rejected record stays in the log (conversation trail), new proposal is `proposed`.

**Key views touched:** My Wishes panel, rejected state UX, resubmit flow.

### Journey 1D — Editing historic planning

**Context:** Anna realizes she mis-recorded planned hours for January (last month's retro showed the plan was wrong to begin with).

1. Anna scrolls the timeline back to January.
2. Clicks a January cell to edit.
3. **Warning dialog:** *"You are editing historic planning for January 2026. This will affect past reports. Continue?"*
4. She confirms, makes the edit. Change log records: user, timestamp, old value, new value, period.
5. No approval needed for her own project — PM can directly correct her own historic plan.

**Key views touched:** Historic edit warning, change log entry.

## Persona 2 — Line Manager (Linjechef)

**Name used in journey:** Per
**Role:** Line manager for the "Electronics Design" group (12 people)
**Primary questions:**
- Is my team over- or under-booked?
- Who has free capacity I can offer to projects?
- What wishes are waiting for my approval?

### Journey 2A — Capacity overview

**Context:** Per opens the app Monday morning to see the state of his group.

1. **Land on Line Manager Home.** Role switcher: "Linjechef — Per". Group auto-scoped to Electronics Design.
2. **Capacity heatmap** at top: rows are his 12 people, columns are months (current ±N). Each cell is colored by utilization:
   - Green: 60–90% booked
   - Red: >100% (overbooked)
   - Yellow: <60% (underbooked, free capacity available)
   - Grey: absence/vacation
3. Per sees Erik is red in March (117%) and Sara is yellow in June (40%). He now knows who to re-balance.
4. **Approval queue card** in the sidebar: "3 wishes waiting." He clicks it.

**Key views touched:** Line Mgr Home, Group heatmap, Approval queue entry point.

### Journey 2B — Approving / rejecting a wish

**Context:** Anna's wish for 60h of Sara's time in June is in Per's queue.

1. Approval queue opens. Each card shows:
   - Requester (Anna), project (Nordlys), person (Sara), period (June), current value (20h) → requested value (60h), note
   - Impact preview: "Sara's June utilization would go from 40% → 90%"
2. Per clicks **Approve**. Sara's plan updates instantly; a change log entry is written; Anna gets notified (in-app, not email for v5.0).
3. Alternative: Per clicks **Reject**. A reason field is required. He types "Sara has another commitment — can offer 40h max." Anna's wish becomes `rejected` with the reason.
4. Alternative: Per clicks **Counter-propose**. He edits the value (60 → 40) and adds a note. The wish goes back to Anna as a `counter-proposed` state for her to accept or reject. *(Optional for v5.0 — flag this: nice but not critical.)*

**Key views touched:** Approval queue, impact preview, approve/reject/counter actions.

### Journey 2C — Direct edit on own group

**Context:** Per wants to pencil in a planning scenario where he moves 20h of Erik's time from Nordlys to another project for April.

1. Per goes to a full timeline view of his group (rows: people, cols: months, cells broken down by project).
2. Clicks Erik / April / Nordlys cell, edits directly — no approval gate, because he owns the group.
3. Change is logged but goes live immediately. Anna (PM for Nordlys) will see the change surfaced in her change feed next time she opens the app.

**Key views touched:** Group timeline with project breakdown, direct edit, change log.

### Journey 2D — Uploading actuals from Excel

**Context:** Per has last month's time dump from the company's shared Excel tracker and wants to upload it.

1. Per goes to **"Import Actuals"** in the admin/tools menu.
2. Sees a brief description, a link to download the template, and an upload dropzone.
3. Drops the file. App parses and shows a **preview table**: rows, detected columns, any parse warnings (bad dates, unmatched person names, etc.).
4. Checkbox option: *"Skriv över manuella ändringar"* — unchecked by default.
5. Summary: "342 rows detected — 18 new, 320 updates, 4 warnings. Click **Import** to proceed, or **Cancel**."
6. Per clicks Import. Progress bar. Confirmation: "Import complete. Batch ID: 2026-04-07-001."
7. If something went wrong: **"Rollback this import"** button stays available for 24h (or until next import).

**Key views touched:** Import entry, template download, upload + preview, override checkbox, rollback.

## Persona 3 — Individual Staff Member (Medarbetare)

**Name used in journey:** Sara
**Role:** Engineer in Electronics Design
**Primary questions:**
- What am I supposed to be working on this month / next month?
- Is my allocation realistic?

### Journey 3A — Checking my schedule

1. **Land on Staff Home — "My Schedule".** Role switcher: "Medarbetare — Sara".
2. Simple view: rows are the projects Sara is allocated to, columns are months (current ±N).
3. Each cell shows planned hours. If actuals exist, shows plan | actual split.
4. A summary strip at top: "This month: 160h planned / 112h logged so far (70%)."
5. No edit controls, no admin tools. Completely read-only.
6. Sara can click a month header to see a daily breakdown (still read-only) — nice-to-have.

**Key views touched:** Staff Home, read-only schedule, plan-vs-actual split.

*That's it for staff. Deliberately minimal — scope discipline.*

## Persona 4 — R&D Manager (R&D-chef)

**Name used in journey:** Karin
**Role:** R&D manager — portfolio oversight across all projects and groups
**Primary questions:**
- How is the whole portfolio tracking against plan?
- Are we over-committed anywhere?
- Which projects are burning ahead/behind?

### Journey 4A — Portfolio overview

1. **Land on R&D Manager Home.** Role switcher: "R&D-chef — Karin". No scope filter — sees everything.
2. **Portfolio grid at top:** rows are all projects, columns are months. Each cell shows total planned hours (summed across all people on that project).
3. Color cues for plan-vs-actual delta at project level — green on plan, red over, yellow under.
4. **Secondary view toggle:** swap project-rows for group-rows, to see which line groups are carrying which load.
5. **Drill into any project** → opens the same timeline view the PM sees (read-only for Karin, or editable if she has admin rights).
6. **Long-horizon mode:** Karin can zoom out to 20–30 months forward. This is where the 53-week / multi-year rendering needs to work cleanly.

**Key views touched:** Portfolio grid, project/group view toggle, long-horizon zoom, cross-drill into PM view.

### Journey 4B — Spotting an overcommit

1. Karin sees a red cell: "May 2026 — total planned 12,400h; line group capacity 11,800h." Overcommit of 600h.
2. Clicks the cell → **breakdown dialog**: which projects contribute, which people are most overbooked.
3. From the dialog, she can navigate to a specific person-month to see who needs re-balancing.
4. Karin doesn't edit directly — she flags it to the relevant line manager (in-app comment or just a phone call in v5.0).

**Key views touched:** Overcommit drill-down, cross-role navigation.

## Persona 5 — Admin (Janne)

**Name used in journey:** Janne
**Role:** Admin (per client request, item #16)
**Primary questions:**
- Are people, projects, departments, and disciplines in the system up to date?
- Who changed what and when?

### Journey 5A — Adding a new engineer to the register

**Context:** The client onboarded a new electronics engineer named Lisa. Janne needs to add her so line manager Per can plan against her capacity.

1. **Land on Admin Home / Change Log feed.** Role switcher: "Admin — Janne". Default landing is the change log so Janne sees recent activity at a glance.
2. Janne clicks **"Register" → "People"** in the admin nav.
3. Sees a table of all current people (active by default) with columns: Name, Department, Discipline, Target hours/month, Created.
4. Clicks **"Lägg till"** (top-right). A side sheet opens with a form: First name, Last name, Department (dropdown), Discipline (dropdown), Target hours/month (default 160).
5. Fills in Lisa's details, clicks **Save**.
6. Toast: "Person added." Row appears in the table; change_log records `REGISTER_ROW_CREATED` with the snapshot.
7. Per (line manager) on his next visit sees Lisa in the Electronics Design heatmap with 0% utilization, ready to be allocated.

**Key views touched:** S14 admin register table, side-sheet form, change_log entry creation.

### Journey 5B — Archiving an unused project

**Context:** "Forsen" was completed last quarter and Janne wants it out of the active project list.

1. Janne navigates to **Register → Projects**, finds Forsen.
2. Row hover reveals an **Archive** button.
3. Clicks Archive. Confirmation dialog: "Arkivera projekt Forsen? Det kommer att döljas från alla aktiva vyer men finns kvar i historiken."
4. Confirms. If Forsen has any active allocations, the dialog instead shows an error listing the dependents: "Detta projekt har 4 aktiva tilldelningar. Avsluta dem först." (Per ConflictError DEPENDENT_ROWS_EXIST.)
5. If no dependents: row gets an "Arkiverad" badge, hides from default view, change_log records `REGISTER_ROW_DELETED`.
6. Toggle "Visa arkiverade" surfaces the row again for audit.

**Key views touched:** S14 admin register table, archive flow, dependent-row blocking, change_log entry.

## Cross-cutting UX Primitives

These appear in multiple journeys and should be built as reusable components:

1. **Timeline grid** — rows × month columns, cells with plan / actual / split rendering. Used in PM timeline, Line mgr group view, Staff schedule, R&D portfolio.
2. **Plan-vs-actual cell** — shows planned, shows actual (if any), shows delta color. Same component everywhere.
3. **Inline cell edit with state** — supports `draft` / `proposed` / `approved` / `rejected` rendering and inline wish submission.
4. **Drill-down drawer** — right-side drawer that shows daily breakdown for a person-project-period. Shared across personas.
5. **Change log feed** — filterable by project, person, period, or author. Different personas see differently scoped defaults.
6. **Approval queue card** — used on Line Mgr Home and potentially as a badge on other persona home screens.
7. **Historic edit warning dialog** — fires on any edit to a period before the current month.
8. **Role switcher** — top-level header control, changes scope and default views.
9. **Import wizard** — template, dropzone, preview, override checkbox, confirmation, rollback. Used for actuals and potentially plan imports later.
10. **Long-horizon zoom** — month/quarter/year zoom levels on the timeline grid; must handle ISO 8601 and 53-week years correctly.

## Explicit Non-Goals for v5.0

To keep scope honest:

- No real authentication — role switcher only
- No task/activity sub-dimension under projects
- No multi-entry-per-day preservation (sum on import)
- No notifications outside the app (email, Slack, etc.)
- No staff actuals entry — read-only for staff
- No hard locks on historic edits
- No counter-proposal flow (flagged optional, defer unless client pushes)
- No drag-and-drop reorder of projects/people in views (drag-to-copy planned hours IS in scope — see client note #5)
- No mobile-first design — desktop primary, mobile acceptable degradation
