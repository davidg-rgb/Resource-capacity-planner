## Framing: Three Personas

The core insight from this round is that different roles need different *views* of the same underlying plan. v5.0 should be organized around persona-specific journeys, not just features.

### Persona 1 — Project Manager (Projektledare)
**Goal:** See and manage planning for *their* specific project(s).
**Key needs:**
- View planned hours across their project timeline
- Propose planning changes (wishes/requests) for their project
- Submit requests to line managers for approval
- Compare planned vs. actual hours to understand project health

### Persona 2 — Line Manager (Linjechef)
**Goal:** See how *their staff* is distributed and what capacity is available.
**Key needs:**
- View planned hours for their line group across projects and months
- Approve/reject project manager requests before they become "live" plan
- Spot over-/under-allocated staff
- Historic view: what did we actually deliver vs. plan?

### Persona 3 — Individual Staff Member (Medarbetare)
**Goal:** See their own schedule and upcoming work.
**Key needs:**
- Personal view of planned hours across projects
- Read-only (or limited) — they consume the plan, not edit it
- Potentially: see own actuals vs. plan

### Bonus Persona — R&D Manager (R&D-chef)
Cross-cutting view: planned time across all projects and months (portfolio-level).

---

## Key Architectural Theme: Plan vs. Actual

A major v5.0 addition is the **actuals layer** — a separate data dimension alongside planned hours:

- Upload/input actual hours worked (per person, per project, per period)
- Import time dumps from external systems (time tracking tools)
- Import must be **idempotent** — repeat imports should not create duplicates (needs an index/unique key: person + project + period)
- Historic planned data must be **editable** (not frozen) so corrections are possible
- Side-by-side comparison: estimated vs. actual, both backward and forward

This is a structural change — likely a new `actuals` table/dimension, plus import pipeline, plus comparison views on every relevant widget.

---

## Second Theme: Request → Approval Workflow

Currently plans are edited directly. Client wants a **proposal layer**:

1. Project manager enters *wishes* (proposed changes) for their project
2. Line manager reviews the proposal
3. On approval, the wish becomes part of the live plan
4. Before approval, wishes are visible but distinct from committed plan

This means plan entries need a state: `draft` / `proposed` / `approved`, and probably an audit trail.

## Proposed v5.0 Theme Groups

Rough grouping for milestone planning — order reflects dependency, not priority:

1. **Data model: Actuals layer** — new dimension, storage, editability of historic plan (items 1, 7)
2. **Import pipeline** — idempotent time-dump import with unique index (items 8, 9)
3. **Persona-based views / role-aware dashboards** — PM, Line mgr, Staff, R&D mgr views (items 10–13)
4. **Proposal → approval workflow** — plan entry states, PM wishes, line mgr approval (item 14)

## Decisions Locked In

### Auth & roles (Q1) — RESOLVED
**Decision:** Option (c) — role-switcher in header, no real auth for v5.0.
**Rationale:** Personas are UX shortcuts, not security boundaries. There are no secrets between PM / Line mgr / R&D mgr use cases; individual staff is the only role with privacy concerns, and staff is not the primary buyer. Job-to-be-done: "when a line manager opens the app, the view they need is right there — no hunting through filters."

### Actuals granularity (Q2) — RESOLVED
**Decision:** Store at **day grain**; allow input at **day / week / month**; display aggregated by user choice (default month).
**Rationale:** External time dumps almost always export per-day — storing at month would throw away information at import time. Storing raw and aggregating on read is cheap (~70k rows/year for this scale) and keeps doors open for future daily drill-downs.

**Input grain rule:** When input is week or month, distribute evenly across working days in that period and store as daily rows. Everything downstream works uniformly on daily data.

**Conflict rule (default):** Last write wins. Flag conflicts when a re-upload would overwrite manually-edited days. Revisit when designing import pipeline.

### ISO 8601 / 53-week year — ARCHITECTURE REQUIREMENT
Sweden uses **ISO 8601 week numbering** (Monday-start, week 1 = week containing first Thursday). This has hard technical implications:

- **53-week years exist** — 2026, 2032, 2037 are the next ones. **v5.0 ships into 2026, a 53-week year.** This is not a theoretical edge case; we'll hit it immediately.
- Week 53 straddles the year boundary (contains days from both Dec and Jan) — matches the client's "break period between years" framing.
- **Never** use JS native `Date.getDay()` defaults — they're US (Sunday-start). Use `date-fns` with `{ weekStartsOn: 1 }` and `getISOWeek()` / `getISOWeekYear()`, or Luxon.
- **Excel imports:** Excel's `WEEKNUM()` defaults to US; `ISOWEEKNUM()` is SE-correct. Any Excel-sourced import must clarify which was used, or we validate by cross-checking a known date.
- **Display format:** Swedish convention is `v.14` or `vecka 14, 2026`. Standardize across UI.

Architect and import pipeline must both treat ISO 8601 + 53-week as first-class, not a locale patch.

### Import pipeline & unique key (Q3) — RESOLVED
**Source format:** Excel (.xlsx) is the primary format — assume client is currently managing everything in Excel spreadsheets. CSV as secondary. Import pipeline must handle messy human-edited spreadsheets (merged cells, inconsistent headers, locale-dependent date formats), not clean API exports. Ship a downloadable Excel template.

**Primary unique key:** `(person_id, project_id, date)` — project-level grain, no task/activity sub-dimension for v5.0.

**Multi-entry handling:** Sum multiple entries per person-project-day at import time (lossy, but project-level grain doesn't need per-entry detail).

**Audit columns:**
- `import_batch_id` — for rollback and traceability
- `source` — `'import'` / `'manual'`

**Override checkbox at upload:** "Skriv över manuella ändringar" — unchecked by default (protective of manual edits), user opts in for bulk corrections.

### Approval workflow (Q4) — RESOLVED
**Decision:** Hybrid — PM→line mgr approval workflow for PM wishes, plus universal change log for all edits.

**Plan entry states:** `draft` / `proposed` / `approved` / `rejected`. Visual treatment at design time (suggested: pending badge, dashed border, or muted color for `proposed`).

**Ownership model:**
- PMs submit *wishes* for their projects → require line mgr approval before going live
- Line managers edit their group's plan directly (no approval gate)
- R&D manager and admin edit freely
- Every edit — regardless of path — lands in the change log

**Rejection path:** Rejected wishes persist with reason. PM can edit and resubmit. Builds a conversation trail, no silent failures.

**Change log:** Universal — captures who/when/what for every edit. Doubles as the audit trail for editable history (Q6).

### Staff access level (Q5) — RESOLVED
**Decision:** Read-only "My Schedule" view.

Staff see their own planned hours grouped by project and month, with the same plan-vs-actual comparison other roles get. No editing, no actuals entry, no change requests. Rationale: primary buyer is line mgr / PM / R&D mgr; going beyond read-only pulls us into time-tracking territory and competes with the Excel workflow the import pipeline is designed to consume.

### Historic edit rules (Q6) — RESOLVED
**Decision:** Editable with soft warning, no hard locks, no mandatory reasons.

- Any edit to a period **before the current month** triggers a confirmation dialog: "You're editing historic planning for [period]. This will affect past reports. Continue?"
- Change log (Q4) automatically captures who/when/what — effectively gives us an audit trail without forcing users to type reasons
- No locking when actuals exist — client explicitly wants to edit history, and legitimate reasons exist (corrections, restructuring, data cleanup)
- **Threshold:** "before current month" is the default. Quarter-based threshold noted as a fallback if monthly proves too noisy in practice.
