## 1. Executive Summary

v3.0 is a UX-only milestone. No new database tables, no new API endpoints (except one lightweight role-resolution endpoint for landing routing), no new business logic. The goal: make the existing features self-explanatory enough that a Swedish line manager switches from Excel on day one.

**Core changes:**
- Role-based landing pages (heat map as front door for planners, dashboard for admins)
- Task-oriented Swedish/English navigation labels with descriptions
- Heat map enhancements (overload sorting, summary banner, click-to-edit)
- Import-to-value flow (import → heat map redirect with success context)
- i18n framework (next-intl) with Swedish as default locale

**Scope guardrails:** No new features. No mobile optimization. No new charts/widgets. No enterprise features. Only modify existing components and add thin routing/i18n layers.

## 3. Architecture Decision Records

### ADR-V3-01: Server-Side Role Resolution for Landing Page

**Decision:** Resolve user role in the server component `(app)/page.tsx` using Clerk's `auth()` and use Next.js `redirect()` to route to the appropriate landing page.

**Alternatives:**
1. Client-side redirect via `useAuth()` + `router.replace()` — causes flash of `/input` before redirect
2. Middleware redirect in `proxy.ts` — adds complexity to auth middleware, harder to maintain
3. Server component redirect in `(app)/page.tsx` — clean, no flash, uses existing patterns

**Rationale:** The app already uses server-side `redirect()` for onboarding (in `(app)/layout.tsx`). Same pattern for role-based landing. Zero client-side flash. The `auth()` call is already happening in the layout, so role data is available.

### ADR-V3-02: Swedish-First Navigation Labels (Pre-i18n)

**Decision:** Phase 19 changes nav labels to Swedish task-oriented names with English descriptions. Phase 22 migrates to next-intl for full i18n.

**Alternatives:**
1. Wait for i18n before changing labels — delays the most impactful UX fix
2. English task-oriented labels first, then Swedish — wrong language for actual users
3. Swedish labels now, i18n later — delivers value immediately, refactors cleanly

**Rationale:** The actual users are Swedish line managers. English nav labels are a real usability barrier today. Hardcoding Swedish labels in Phase 19 delivers immediate value. Phase 22 wraps them in next-intl for proper localization. The refactor from hardcoded strings to message keys is straightforward.

### ADR-V3-03: URL Params for Import-to-Heat-Map Context

**Decision:** After successful import, redirect to `/dashboard/team?imported=N&source=import`. The heat map page reads these params and shows a contextual success banner.

**Alternatives:**
1. Sonner toast — disappears after 5 seconds, user might miss it
2. React state/context — lost on page navigation
3. URL search params — persistent, bookmarkable, survives refresh, easy to implement

**Rationale:** URL params are the simplest mechanism that works with Next.js server components. The heat map page can read `searchParams` on the server side and render a banner. After the user dismisses or navigates away, the params are gone.

### ADR-V3-04: next-intl for Internationalization

**Decision:** Use next-intl for i18n with Swedish (`sv`) as default locale and English (`en`) as fallback.

**Alternatives:**
1. react-i18next — widely used but less App Router integration
2. next-intl — purpose-built for Next.js App Router, supports server components, type-safe
3. Custom solution — unnecessary complexity for a well-solved problem

**Rationale:** next-intl has first-class Next.js App Router support, works in both server and client components, and provides type-safe message access. The library is actively maintained and handles locale detection, routing, and message formatting.

### ADR-V3-05: Heat Map Sort Strategy

**Decision:** Sort heat map rows within each department group by max utilization (descending) — overloaded people float to the top of their department section.

**Alternatives:**
1. Global sort ignoring departments — breaks the department grouping UX that already works
2. Sort within department groups — preserves grouping while surfacing problems
3. Separate "overloaded" section at top — requires restructuring the heat map data model

**Rationale:** Department grouping is a key navigation pattern for managers who think in terms of their team structure. Sorting within groups preserves this mental model while making overloaded people immediately visible. The sort happens client-side on the already-fetched data — no API changes needed.

## 5. Module & Function Definitions

### 5.1 Role-Based Landing — `src/app/(app)/page.tsx`

**Purpose:** Resolve the authenticated user's Clerk role and redirect to the appropriate landing page.

**Current behavior:** `redirect('/input')` for all users.

**New behavior:**

```
getRoleLandingPage(orgRole: String | null): String
  Purpose: Map a Clerk organization role to the correct landing URL
  Params:
    - orgRole: Clerk role string (e.g., 'org:admin', 'org:planner', null)
  Returns: URL path string
  Mapping:
    - 'org:owner'   → '/dashboard'
    - 'org:admin'    → '/dashboard'
    - 'org:planner'  → '/dashboard/team'
    - 'org:viewer'   → '/dashboard/team'
    - null / unknown → '/dashboard/team'
  Side effects: None (pure function)
  Error cases: Unknown role falls back to '/dashboard/team' (heat map is the safest default — it's read-only and shows immediate value)
  Called by: Home page server component (page.tsx default export)
  Calls: None
```

```
Home(): ServerComponent
  Purpose: Root page that redirects authenticated users to their role-appropriate landing
  Params: None (Next.js page component)
  Returns: Never (always redirects)
  Side effects: Calls redirect() which throws internally
  Error cases:
    - No auth session → Clerk middleware handles (redirects to /sign-in)
    - No orgRole → defaults to '/dashboard/team'
  Called by: Next.js router when user navigates to '/'
  Calls: auth() from @clerk/nextjs/server, getRoleLandingPage(), redirect() from next/navigation
```

**Implementation detail:** The `auth()` call returns `{ orgRole }` which is a Clerk role string like `'org:admin'`. This is available in server components without any database query.

**Acceptance criteria:**
- [ ] AC-01: User with `org:planner` role lands on `/dashboard/team` after login

### 5.5 Heat Map Summary Banner — `src/components/heat-map/heat-map-summary-banner.tsx` (NEW)

**Purpose:** Show a prominent banner at the top of the heat map page summarizing overload status.

```
HeatMapSummaryBanner(props: HeatMapSummaryBannerProps): ClientComponent
  Purpose: Display an at-a-glance overload summary above the heat map table
  Props:
    - data: HeatMapResponse — The full heat map data (departments with people and months)
  Returns: A banner div showing overload count and most critical month, or nothing if no overloads

  Rendering logic:
    1. Iterate all people across all departments
    2. For each person, find months where status === 'over' (hours > targetHours)
    3. Count distinct overloaded people
    4. Find the month with the highest total overload hours
    5. Render:
       - If overloadedCount > 0:
         Red/warning banner:
         "{N} {personer/person} överbelastade — mest kritiskt: {monthLabel}"

         Green/success banner:
         "Inga överbelastade medarbetare"

  Side effects: None (pure rendering)
  Error cases:
    - Empty data (no departments) → render nothing
    - All people are idle → show green "Inga överbelastade"
  Called by: TeamOverviewContent (dashboard/team/page.tsx)
  Calls: calculateHeatMapStatus() from src/lib/capacity.ts
```

### 5.7 Heat Map Table Sort — `src/components/heat-map/heat-map-table.tsx` (MODIFY)

**Purpose:** Sort people within each department group by maximum utilization (descending), so overloaded people appear first.

```
sortPeopleByUtilization(people: List<HeatMapPerson>): List<HeatMapPerson>
  Purpose: Sort people within a department group so overloaded appear first
  Params:
    - people: Array of people with their monthly allocation data
  Returns: New sorted array (does not mutate input)
  Sort key: Maximum utilization ratio across all visible months (descending)
  Algorithm:
    For each person:
      maxRatio = max(person.months.map(m => m.hours / m.targetHours))
    Sort by maxRatio descending (highest utilization first)
  Side effects: None
  Error cases:
    - targetHours is 0 for all months → treat ratio as 0 (idle people sort last)
  Called by: HeatMapTable render loop (inside department map)
  Calls: None
```

```
HeatMapTable (MODIFIED behavior)
  Change: Before rendering people within each department, apply sortPeopleByUtilization()
  Current: People rendered in order received from API
  New: People sorted by max utilization descending within each department

  Additional change: Person names are rendered as clickable links
  Current: Plain text with initials avatar
  New: <Link href="/input/{personId}"> wrapping the person name

  Called by: TeamOverviewContent
  Calls: sortPeopleByUtilization() (new), Link from next/link
```
