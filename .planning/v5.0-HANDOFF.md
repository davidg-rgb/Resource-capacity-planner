## Key constraints to remember

- **Strictly additive to v4.0.** No migrations that touch existing tables except adding `projects.lead_pm_person_id`. Existing `allocations` table (monthly grain) is untouched and keeps serving v4.0 screens.
- **Personas are UX shortcuts, not security boundaries** (ADR-004). No real auth in v5.0 — role-switcher in header only.
- **2026 is a 53-week ISO year.** All date logic routed through `lib/time/iso-calendar.ts` (Phase 0.1 must build first); no other file may import `date-fns` or use native `Date` locale defaults.
- **Swedish holidays hardcoded for 2026–2030** in iso-calendar; values are in §6.1 of the architecture doc.
- **Day-grain actuals storage, monthly-grain plans.** Actuals input accepted at day/week/month, distributed via largest-remainder method (ADR-010), stored daily.
- **Universal change_log** — every mutating service writes via `recordChange`, enforced by an eslint rule + a separate codegen script + a runtime test (TC-CL-005).
- **Excel is the import format.** SheetJS (`xlsx` package) already in deps. Two layouts: row-per-entry (canonical) and pivoted. US-week headers are a hard-stop ValidationError.
