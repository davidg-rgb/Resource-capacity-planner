## Option 3 — Schedule recurring pre-ship audit

**Why this matters:** the audit team caught commit `27ac599` ("feat(51-02): dashboard_layouts JSONB migration") which silently rewound Phase 49 + Phase 50 work — wiping the persona-keyed sidebar (NAV-02), persona-aware breadcrumbs (NAV-03), department picker (UNBREAK-08), and grouped persona switcher (NAV-04). v6.0 shipped with all four broken. Tag `v6.0` is on the regressed code; only the audit caught it. **Without a routine pre-ship audit, the next milestone is one bad merge away from the same outcome.**

**Two ways to schedule:**

### 3a — One-time audit before each milestone close (manual gate)

Add a step to `.planning/PROJECT.md` "Evolution" section:
> Before running `/gsd-complete-milestone v{X.0}`, run a 4-round code-vs-architecture audit (R1: 4 parallel scanners, R2-R5 verification + fixes until 0 P0/P1).

Pros: cheap, explicit, owner-controlled.
Cons: relies on memory.

### 3b — Recurring background agent (autonomous)

Trigger this with:

```
/schedule create
- name: pre-ship-audit
- cadence: every 2 weeks
- task: "Run a 1-round code-vs-architecture audit. Spawn 4 parallel scanners (ARCHITECTURE.md / v5.0-ARCHITECTURE.md / current-milestone-plan / gsd-code-reviewer). If 0 P0+P1 findings, post a one-line OK to chat and exit. If P0+P1 found, post the consolidated findings file path and stop without fixing. The user decides whether to fix in a separate session."
```

Pros: autonomous, catches drift between milestones, doesn't auto-fix (you stay in control of what merges).
Cons: agent runs on a cadence not tied to actual ship events — may scan a quiet branch.

### 3c — On-PR audit (best-but-most-invasive)

Wire a GitHub Action that runs the audit on every PR to `main`. Each scanner becomes a CI job; merge blocked on `0 P0`. This is the "real" answer but takes 1-2 phases of work to wire properly.

**Recommended:** start with 3b for now (cheapest, catches the regression class). Move to 3c if 3b finds enough drift to be worth automating.

## Lessons from this session worth saving

These three patterns are worth remembering for future sessions:

1. **Silent regressions in unrelated commits.** `27ac599` was titled "JSONB migration" and reviewers waved through the dashboard_layouts SQL. Three persona files were silently rewound to pre-v6.0 state in the same commit — nobody noticed until the audit. Pattern: **a commit's title is not a contract**; the diff is. Routine audits catch this; pre-merge `git diff --stat` checks against the commit's stated scope can too.

2. **Tests that pass on broken code via mock.** D-CR-01 (PM `homeDepartmentId` never populated) — unit tests passed because they constructed the persona directly with `homeDepartmentId: 'dept-A'`. Production switcher never set it. **Integration tests that go through the real composition root would have caught it.** Going forward: at least one integration test per feature should exercise the actual user-facing entry point, not the unit under test.

3. **Doc drift is normal, but ARCHITECTURE.md baseline header is the right pattern.** The v1.0 ARCHITECTURE.md was 4708 lines of stale spec by v6.0. The R2 fix added a 30-line baseline header at the top: "this is v1.0; for current state see X, Y, Z" + drift notes for the worst stale entries. This is cheaper than rewriting the doc and clearer than letting it rot. Apply the same pattern to v5.0-ARCHITECTURE.md when v8.0 starts diverging from it.
